# HTTP Exceptions

Implements a number of Python exceptions which can be raised from within
a view to trigger a standard HTTP non-200 response.

:target{#usage-example}

## Usage Example

```python
from werkzeug.wrappers.request import Request
from werkzeug.exceptions import HTTPException, NotFound

def view(request):
    raise NotFound()

@Request.application
def application(request):
    try:
        return view(request)
    except HTTPException as e:
        return e
```

As you can see from this example those exceptions are callable WSGI
applications. However, they are not Werkzeug response objects. You
can get a response object by calling <code>get\_response()</code> on a HTTP
exception.

Keep in mind that you may have to pass an environ (WSGI) or scope
(ASGI) to <code>get\_response()</code> because some errors fetch additional
information relating to the request.

If you want to hook in a different exception page to say, a 404 status
code, you can add a second except for a specific subclass of an error:

```python
@Request.application
def application(request):
    try:
        return view(request)
    except NotFound as e:
        return not_found(request)
    except HTTPException as e:
        return e
```

## Special HTTP Exceptions

Starting with Werkzeug 0.3 some of the builtin classes raise exceptions that
look like regular python exceptions (eg [<code>KeyError</code>](https://docs.python.org/3/library/exceptions.html#KeyError "(in Python v3.11)")) but are
[<code>BadRequest</code>](#werkzeug.exceptions.BadRequest "werkzeug.exceptions.BadRequest") HTTP exceptions at the same time.  This decision was made
to simplify a common pattern where you want to abort if the client tampered
with the submitted form data in a way that the application can’t recover
properly and should abort with <code>400 BAD REQUEST</code>.

Assuming the application catches all HTTP exceptions and reacts to them
properly a view function could do the following safely and doesn’t have to
check if the keys exist:

```default
def new_post(request):
    post = Post(title=request.form['title'], body=request.form['body'])
    post.save()
    return redirect(post.url)
```

If <cite>title</cite> or <cite>body</cite> are missing in the form, a special key error will be
raised which behaves like a [<code>KeyError</code>](https://docs.python.org/3/library/exceptions.html#KeyError "(in Python v3.11)") but also a [<code>BadRequest</code>](#werkzeug.exceptions.BadRequest "werkzeug.exceptions.BadRequest")
exception.

## Custom Errors

As you can see from the list above not all status codes are available as
errors.  Especially redirects and other non 200 status codes that do not
represent errors are missing.  For redirects you can use the <code>redirect()</code>
function from the utilities.

If you want to add an error yourself you can subclass [<code>HTTPException</code>](#werkzeug.exceptions.HTTPException "werkzeug.exceptions.HTTPException"):

```default
from werkzeug.exceptions import HTTPException

class PaymentRequired(HTTPException):
    code = 402
    description = '<p>Payment required.</p>'
```

This is the minimal code you need for your own exception.  If you want to
add more logic to the errors you can override the
<code>get\_description()</code>, <code>get\_body()</code>,
<code>get\_headers()</code> and [<code>get\_response()</code>](#werkzeug.exceptions.HTTPException.get_response "werkzeug.exceptions.HTTPException.get_response")
methods.  In any case you should have a look at the sourcecode of the
exceptions module.

You can override the default description in the constructor with the
<code>description</code> parameter:

```default
raise BadRequest(description='Request failed because X was not present')
```
